# 3.6 ตอบคําถามต่อไปนี้เพื่อวัดความเข้าใจ

## 1. 'input' vs 'change' event: ต่างกันอย่างไร? ทําไมต้องใช้ 'input' สําหรับ real-time validation

> input event: จะทำงาน ทันที ทุกครั้งที่มีการเปลี่ยนแปลงค่าใน input
> change event: จะทำงานก็ต่อเมื่อผู้ใช้กด Enter คือต้องพิมพ์เสร็จแล้วและคลิกออกจากช่องนั้นก่อน Event ถึงจะทำงาน

> ทำไมถึงใช้เพราะว่า เพื่อให้เกิด Real-time validation คือระบบจะตรวจสอบความถูกต้องและแสดงสีเขียว/แดงให้ผู้ใช้เห็นทันทีขณะที่กำลังพิมพ์ โดยไม่ต้องรอให้ผู้ใช้คลิกไปที่อื่น ช่วยให้ผู้ใช้แก้ไขข้อมูลได้ทันท่วงที

## 2. preventDefault(): ทําอะไร? ถ้าไม่มี code นี้จะเกิดอะไร

> ทำหน้าที่ใช้สำหรับหยุดพฤติกรรมเริ่มต้น ของ Browser

> ถ้าไม่มีcodeนี้ เมื่อกดปุ่ม Submit Browser จะทำการ Refresh หน้าเว็บ เพื่อส่งข้อมูลไปยัง Server ตามปกติของ HTML Form
> ผลที่ตามมาคือ JavaScript ที่เราเขียนไว้เพื่อตรวจสอบความถูกต้อง จะแสดงผลแล้วหายไปทันทีเพราะหน้าเว็บถูกรีโหลดใหม่

## 3. classList operations: ทําไมต้อง remove('error') และ add('valid') ทั้งคู่? ไม่ได้มี add('valid') คนเดียวไหม

> หากใช้แค่ add('valid') โดยไม่ลบ error ออก จะทำให้ element นั้นมีทั้งสองคลาสพร้อมกัน คือ <input class="error valid">
> ซึ่งจะทำให้การแสดงผลผิดพลาด
> จึงต้องใช้ remove('error') เพื่อล้างสถานะเก่าที่เป็นสีแดงออกก่อน แล้วค่อยใส่สถานะใหม่ add('valid') เพื่อให้สมบูรณ์

## 4. disabled attribute: submitBtn.disabled = true หมายความว่าอะไร? สามารถจัดการผ่าน JavaScript ได้ไหม

> หมายความว่าปุ่มนั้นจะถูก ปิดการใช้งาน ผู้ใช้ไม่สามารถคลิกปุ่มนั้นได้
> สามารถจัดการได้ เราสามารถใช้ JavaScript เพื่อตรวจสอบสถานะของฟอร์ม (isFormValid()) และสั่งเปิด/ปิดปุ่มได้แบบDynamic

## 5. Email validation: value.includes('@') && value.includes('.') กรณีไหนจะผิด

> o "user@gmail..com" (2 dots) - ผ่านหรือไม่
> ผ่าน เพราะใน text มีทั้งเครื่องหมาย @ และ . ครบตามเงื่อนไข
> o "user.gmail.com" (ไม่มี @) - ผ่านหรือไม่
> ไม่ผ่าน เพราะขาดเครื่องหมาย @ ทำให้เงื่อนไขเป็น false

## 6. DRY principle: ทําไม code ที่เพิ่ม listener 4 ครั้ง (สําหรับ 4 inputs) ทําให้ code ยาว

> เป็นการทำซ้ำ logic เดิม ทำให้ Code ยาวเกินความจำเป็นและแก้ยาก แก้ไขได้โดยใช้ Loop หรือ querySelectorAll เพื่อเลือก input ทั้งหมดแล้ววนลูป

## 7. confirm-password field: ทําไมต้องมี? ถ้าลบออกจะเกิดปัญหาไหม

> ทำให้ผู้ใช้มองไม่เห็นว่าพิมพ์อะไรลงไปเพื่อควาปลอดภัย
> ถ้าลบออกผู้ใช้อาจพิมพ์รหัสผ่านผิดโดยไม่รู้ตัวเมื่อสมัครสมาชิกเสร็จแล้ว ผู้ใช้จะ Login ไม่ได้เพราะรหัสผ่านที่ระบบบันทึกไปนั้นผิด
